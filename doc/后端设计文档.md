 # 后端技术设计文档

  项目：blog-2-video Hybrid Pipeline 重构
  版本：v1.1
  日期：2026-04-13
  状态：Draft

  ## 1. 目标

  将当前后端从“原始博文直接生成视频”的单阶段链路，重构为明确的两阶段流水线：

  原始博文 / 文章
    -> Script Workflow
    -> 口语化视频文案
    -> Animation Compiler Workflow
    -> Scene / Marks / Layout / Motion / DSL / Code
    -> Remotion 预览与渲染

  核心目标：

  - 先生成适合“讲述”的口语稿，再进入视频编译
  - 将视频生成主链路改造成确定性 compiler
  - 所有中间产物结构化、可回放、可调试
  - scene 级失败隔离、repair、重编译可用
  - Agent 只做增强，不主导结构

  ## 2. 核心原则

  最关键的修正：

  source_text 不是动画编译输入，oral_script 才是动画编译输入。

  也就是说：

  - 博文适合阅读
  - 口语稿适合讲述
  - 场景规划、镜头节奏、marks、layout、motion 都必须基于口语节奏，而不是基于原始博文段落直接生成

  ## 3. 非目标

  本设计不覆盖：

  - 商业化渲染平台
  - 多租户系统
  - 自动风格推荐
  - 分布式任务调度
  - 成片导出平台

  ## 4. 当前系统问题

  现有后端主要问题：

  - 文本改写和视频生成在架构上没有正式衔接协议
  - generate_animation_sse 语义上接受 source_text，实际上容易被直接传入原文
  - animation workflow 缺少明确的上游“口语稿输入约束”
  - Director / Visual / Coder 仍然在直接消费高自由文本
  - 没有形成两阶段状态模型
  - 没有 pipeline 级统一回放和调试视图

  ## 6. 两阶段流水线设计

  ## 6.1 Stage A: Script Workflow

  职责：把博文改造成适合短视频讲述的口语稿。

  输入：

  - source_text

  输出：

  - oral_script
  - script_segments
  - script_metadata
  - review_score
  - feedback

  这一阶段不负责：

  - scene
  - marks
  - layout
  - motion
  - code

  ### 节点设计

  1. parse_source_node
  2. rewrite_oral_script_node
  3. review_oral_script_node
  4. polish_oral_script_node
  5. segment_oral_script_node
  6. finalize_oral_script_node

  职责：把口语稿编译为视频场景与可执行代码。

  - script_segments
  - script_metadata

  输出：

  - scenes
  - marks
  - layouts
  - motions
  - dsl
  - codes
  - validations

  链路：

  source_text

    -> oral_script_result
    -> animation_workflow
    -> animation_result

  输出：

  - script artifacts
  - animation artifacts
  - pipeline summary

  ## 9. Script Workflow 详细设计

  ## 9.1 parse_source_node

  输入：

  - source_text

  职责：

  - 抽取主题、核心观点、语气、目标受众
  - 清洗无关格式噪音

  失败：

  - SCRIPT_SOURCE_INVALID

  ## 9.2 rewrite_oral_script_node

  职责：

  - 将博文改写为可讲述口语稿
  - 保留核心观点，不做视觉设计
  - 句子节奏适配短视频

  失败：

  - ORAL_SCRIPT_REWRITE_FAILED

  ## 9.3 review_oral_script_node

  职责：

  - 评价口语稿是否清晰、自然、可讲述
  - 输出 score 和 feedback

  失败：

  - ORAL_SCRIPT_REVIEW_FAILED

  ## 9.4 polish_oral_script_node

  职责：

  - 根据 feedback 做有限轮次改写
  - 最大循环次数建议 3

  ## 9.5 segment_oral_script_node

  职责：

  - 将口语稿拆成 narration segments
  - 为 scene planning 提供稳定输入

  规则：

  - 以讲述节奏拆分，不以原文段落拆分

  ## 9.6 finalize_oral_script_node

  职责：

  - 固化最终口语稿结果
  - 写入 checkpoint

  ## 10. Animation Workflow 详细设计

  ## 10.1 parse_oral_script_node

  输入：

  - oral_script
  - script_segments

  职责：

  - 将口语 segments 转为 scene planning 可消费格式
  - 明确情绪曲线和讲述段落

  ## 10.2 plan_scenes_node

  职责：

  - 依据口语节奏生成 scene 列表
  - scene 顺序只在此处决定

  失败：

  - SCENE_PLAN_INVALID

  ## 10.3 generate_marks_node

  职责：

  - 根据 scene 时长生成 global marks
  - 根据 scene 内部节奏生成 scene marks

  约束：

  - 时间轴只由本节点决定
  - agent 不得改 global marks

  ## 10.4 compile_layout_node

  职责：

  - 每个 scene 单独生成 layout spec
  - 必须服从手机 safe area

  博文 / 文章 / 原稿
    -> Stage A: 口语化文案工作流
    -> Oral Script / Script Segments
    -> Stage B: 视频编译工作流
    -> Scene / Marks / Layout / Motion / DSL / Code

  核心目标：

  - Stage B 只负责“视频结构编译化”

  ### 2.3 确定性原则

  - 口语稿阶段允许有限 LLM 优化
  - 视频编译阶段必须由 deterministic compiler 主导

  ## 8. Stage A 详细设计

  ## 8.1 parse_source_node

  输入：source_text
  输出：主题、语气、重点、结构意图

  职责：

  - 清洗原始文本
  - 提取核心论点
  - 判断更适合科普、观点、案例还是教程表达

  失败处理：

  - SCRIPT_PARSE_INVALID
  - 中止 Stage A

  ## 8.2 rewrite_oral_script_node

  输入：解析结果 + 原文
  输出：第一版口语稿

  职责：

  - 将书面语改为口语表达
  - 压缩长句
  - 去掉文章式衔接，增强讲述节奏

  约束：

  - 不改原始含义
  - 不凭空增加事实

  ## 8.3 review_oral_script_node

  输入：口语稿
  输出：评分与反馈

  职责：

  - 评估可讲述性
  - 评估节奏、吸引力、清晰度

  ## 8.4 polish_oral_script_node

  输入：口语稿 + feedback
  输出：修订版口语稿

  策略：

  - 最多 N=2 次
  - 达到阈值则停止

  ## 8.5 segment_oral_script_node

  输入：最终口语稿
  输出：script_segments

  职责：

  - 将口语稿切成适合视频 scene planning 的段落
  - 每个 segment 对应一个讲述单位

  ## 8.6 finalize_oral_script_node

  输出最终 oral_script_result

  要求：

  - 必须成为 Stage B 唯一输入
  - 写入 checkpoint
  - 支持独立查询和回放

  ## 9. Stage B 详细设计

  ## 9.1 parse_oral_script_node

  输入：oral_script_result
  输出：用于 scene planning 的结构化脚本

  职责：

  - 读取 script_segments
  - 规范化 role、priority、预计时长

  ## 9.2 plan_scenes_node

  输入：结构化口语稿
  输出：scenes

  要求：

  - scene 顺序只在此阶段决定
  - scene 应基于讲述节奏，而非原文段落形式
  - 每个 scene 绑定一个或多个 script segments

  ## 9.3 generate_marks_node

  输入：scenes
  输出：marks

  要求：

  - 时间只由此处决定
  - 支持固定时长与字数估时模式
  - scene 内 reveal marks 统一生成

  ## 9.4 compile_layout_node

  输入：scenes + marks + theme_profile
  输出：layouts

  要求：

  - scene 独立 layout
  - 强约束手机安全区
  - 越界在本阶段报错

  ## 9.5 compile_motion_node

  输入：scenes + marks + layouts
  输出：motions

  要求：

  - motion 只能引用已存在 node
  - 时间必须依赖 marks
  - 模板化 profile 优先

  ## 9.6 generate_dsl_node

  输入：scene + layout + motion
  输出：dsl

  要求：

  - 先生成中间树结构
  - 禁止直接让模型生成最终 React 代码

  ## 9.7 generate_scene_code_node

  输入：dsl
  输出：codes

  要求：

  - 模板代码拼装优先
  - 用前端现有 helper 契约
  - 最小化自由生成

  ## 9.8 validate_scene_node

  输入：layout + motion + dsl + code
  输出：validations

  检查项：

  - layout overflow
  - motion target missing
  - invalid DSL node
  - render missing
  - unsafe code
  - text too small

  ## 9.9 repair_scene_node

  输入：失败 scene
  输出：修复后的 artifacts

  策略：

  - 每个 scene 最多 repair 2 次
  - repair 失败则保留失败状态
  - 其他 scene 不受影响
